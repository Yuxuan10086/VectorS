## 上下行帧结构（串口 TTL / RS485 / CAN 载荷一致）

| 字段 | 下行（上位机→驱动板） | 上行（驱动板→上位机） |
|------|------------------------|------------------------|
| 帧头 | `0xC5` | `0xC5` |
| 从机地址 | `addr`（0 广播，1–255） | `addr` |
| 功能码 | `code` | `code`（通常与对应指令相同） |
| 指令数据 | 视功能码而定 | 视功能码而定；首字节多为操作结果 |
| 检验码 | `CHECKSUM` | `CHECKSUM` |
| 帧尾 | `0x5C` | `0x5C` |

**校验（8 位）**

`CHECKSUM = (0xC5 + addr + code + 指令数据所有字节) & 0xFF`

## 默认与常用参数

| 项目 | 默认值 / 说明 |
|------|----------------|
| 自定义协议 | 使用自定义协议前需 **关闭 Modbus**（屏幕菜单 MODBUS→失能，或指令 `0x69`） |
| 从机地址 | 出厂 **0x01** |
| 串口 TTL / RS485 | **115200**，8 数据位，1 停止位，无校验 |
| CAN | **500 kbit/s**，仅 **扩展帧**；下行 ID **`0x1001`**；上行本仓库为 **`0000000x`**（x=地址低 4 位，与 `default_can_eff_id` 一致） |
| CAN 单帧数据 | 每条 CAN 数据段最多 **8 字节**，更长指令需拆成多帧连续发送 |
| 供电（手册产品规格） | **12–32 V**（示例常用 **24 V**） |

## CAN 扩展帧 ID（工程约定）

与本仓库驱动对接时的 **帧 ID** 约定如下（均为扩展帧；十六进制书写）：

| 方向 | 扩展帧 ID | 说明 |
|------|-----------|------|
| 下行（上位机→驱动器） | **0x1001** | **统一固定**，所有下发报文均使用该 ID |
| 上行（驱动器→上位机） | **`0000000x`**（本仓库） | **x** 为从机地址 **低 4 位**（与协议帧内 `addr` 的低半字节一致；例如 addr=5 → **`0x00000005`**）。PDF 中有时写作 **0x100x**，实际扩展帧数值等价关注末字节即可；多轴靠不同 **x** 区分应答 |

具体应答使用的 ID 亦可通过指令 **`0x6C`（设置发送 CAN_ID）** 等与手册一致方式修改；接收端需按实际配置的 ID 过滤、组帧。

### 与本仓库 `robot_driver` 的对应关系（实现以本文为准）

| 项目 | 说明 |
|------|------|
| 校验和 | 与上文 **8 位校验** 公式一致，见 `pd42_protocol.hpp` 中 `checksum_from_parts` / `verify_frame_checksum` |
| 上行操作结果 | 与上表「指令数据」首字节一致：**`0x01` 成功**；**`0xE1`–`0xE6`** 为错误码（同「注意事项」第 6 条） |
| CAN 仅扩展帧 | `can_interface` 收包仅处理扩展帧，忽略标准帧 |
| 下行 / 上行 ID | 下行固定 **`0x1001`**；上行本仓库 **`default_can_eff_id(addr) = (addr & 0x0F)`**（29 位扩展 ID，candump 形如 **`00000005`**）；若需其它数值可用 **`0x6C`** 配置驱动发送 ID |
| 构造时 0x6C / 0x04 | `Pd42Motor` 构造时先 **`send_cmd` 0x6C** 将驱动发送 ID 置为 **`default_can_eff_id(motor_id)`**，成功后再 **`send_cmd` 0x04** 保存参数（手册 4.2.4，掉电不丢失） |

## 注意事项（时序与应答）

1. **单条指令内**：从 `0xC5` 到 `0x5C` 须 **连续发送**，字节间不宜超过 **1 个字节时间** 的间隔，否则可能接收失败。  
2. **指令与指令之间**：两条完整帧之间至少间隔 **2 ms**。  
3. **广播地址 0**、**分组广播**：从机 **不应答**。  
4. **总线上无对应从机地址**：**不应答**。  
5. 可通过 **`0x6F` 指令回响** 配置为 **不回显应答**（不需要应答时使用）。  
6. 上行帧指令数据第一字节为操作结果：**`0x01` 成功**；`0xE1`–`0xE6` 为错误码（帧长、帧头尾、校验、功能码、数据等）。

### 急停（0xFC）与清除状态（0xFB）

- **`0xFC`**：立即停止（刹车）。实测若在 **`0xFC` 成功后** 不尽快 **`0xFB` 清除状态**，电机可能 **长时间严重发烫**（驱动侧持续异常工况未清）。
- **`Pd42Motor::stop()`**：先发 **`0xFC`**，应答成功后 **立即** 再发 **`clear_status()`（`0xFB`）**，满足手册「清除堵转、刹车、失能等相关状态」的用法。

### 模式切换（0x62）与通信运动指令（0xF1 / 0xF2）— 实测现象（工程记录）

- **`0x62`** 表示「通信」侧选用哪种运动指令路径：常用 **`0x01`→`0xF1`（速度）**、**`0x00`→`0xF2`/`0xF3`（位置）**（见下文功能码表中 F 区「XX 模式控制」）。二者与 **`0xF1`/`0xF2` 本体**是不同层级：先由 **`0x62` 选对通道**，再发对应的 **`0xF1` 或 `0xF2`**。
- **现象**：由速度相关用法切入 **通信位置**后，电机可能出现 **异常快速转动**；与 **CAN 收包顺序 / 是否先发 `0xF1` 零速** 无必然联系（零速不能从根上消除该现象）。
- **实测结论（固件侧内部目标位置）**  
  切入位置模式后，电机曾在现象上 **持续试图到达某一固定内部目标位置**（例如观测到约 **`11708570`**（协议位置单位，51200 一圈）量级），表现为位置误差极大时的快速跟踪，而非「读错缓存帧」。  
  **规避做法**：切换位置模式前若先发 **`0xFA` 失能（Byte1=1）再使能（Byte1=0）**（本仓库 `enable(false)` → `enable(true)`），则 **不再出现** 上述异常转动——推断 **失能/再使能会复位或刷新驱动内部与位置命令相关的状态**，避免沿用错误的目标位置寄存器。
- **仍建议的工程顺序**  
  1. **`Pd42Motor::set_mode(...)`** 在发 **`0x62` 前** 已 **自动** **`0xFA` 失能再使能**（`enable(false)` → `enable(true)`），与上条规避做法一致，一般 **不必** 再在外层单独 `en(false)/en(true)`。  
  2. 切入位置后尽快发有效 **`0xF2`**，必要时配合手册 **`0xF8`**（位置清零）理解零点与目标关系。
